# 如何高效使用 Agent：构造一个真正适合人机协作的工作环境

各位大家好。

今天我想分享的，不是一些零散的“提示词技巧”，也不是什么神秘的“prompt engineering 秘诀”。我真正想讲的是：**当我们开始长期、高强度地和 agent 一起工作之后，什么样的工作方式，才能真正把它们的价值释放出来。**

我自己这段时间主要使用的是 **Codex** 这一类 agent-oriented workflow，偏好的模型是 **GPT-5.2**，而且我倾向于让它工作在一种比较高自治的模式里。我的整体感受是：**它确实非常强，尤其强在“可以持续自主运行很久”这件事上。**有时候它会显得慢一点，有时候会显得保守一点，但如果你看最终结果，你会发现它真的能在几个小时的持续推进之后，把事情做出来。就这一点而言，我是满意的，所以这也是我目前最优先使用的一类 agent。

而且我想先讲一个我自己越来越强烈的判断：**agent 的真正意义，不只是“更快”，而是把过去很多根本不会启动的工作，变成现在可以启动、可以推进、甚至可以完成的工作。**

以前很多事情没做，不一定是因为它们不重要，也不一定是因为它们做不到，而是因为它们**太费劲**。你明知道它理论上可以做、也知道数字世界里的东西本来就不是不能复现，但一想到要投入那么多时间、那么多体力、那么多心智负担，你就会自动把它“黑箱化”。所谓黑箱化，就是我们不再去打开它、理解它、重做它，而只是接受它已经存在，并且只去碰它最外面的界面。

所以我现在越来越觉得，agent 真正改变的一件事，是它在不断降低“打开黑箱”的成本。很多以前因为“太麻烦”而被放弃的工作，现在变成了“几分钟可以推进、几十分钟可以看到结果、几个小时可以跑通”的事情。这个变化，我认为非常深刻。

## 一、先讲最核心的一点：要让环境鼓励 agent，而不是拖累 agent

如果今天让我把全部经验压缩成一句话，我会说：

**高效使用 agent 的关键，不是你会不会几句漂亮的提示词，而是你有没有为它准备一个“鼓励它发挥”的环境。**

什么叫“鼓励它发挥”的环境？

很简单，就是你要让它在一个**阻力尽可能小**的地方工作。比如：

* 有足够的磁盘空间；
* 有合适的 GPU；
* 有合适的命令行工具；
* 有稳定、合理的网络环境；
* 有清楚的目录结构；
* 有便于共享上下文的文本文件；
* 有可恢复、可追踪、可回滚的版本控制。

如果这些东西没有准备好，agent 不会像人一样先抱怨，它通常会默默地退回到一些更保守、更慢、更笨的方法上。你表面上看它“还在工作”，但其实它已经在一个很差的轨道上浪费很多时间了。

举个很具体的例子。现在很多 agent 的第一反应，其实是很依赖 **ripgrep** 的，也就是命令行里常用的 `rg`。如果你的环境里没有装 `rg`，它未必会主动替你装——因为现在很多 agent 都偏保守，它们不太愿意为了完成任务就主动修改你的系统环境。于是它就会退回去使用更旧、更慢、更不合适的工具。这样一来，每一次搜索、每一次扫描目录、每一次定位文件，都会白白损失时间。

但站在我们的角度，这其实只是几下键盘的事。你是管理员，你知道这台机器该装什么。那你就应该替它把这些基础设施准备好。

我的做法常常是这样的：如果发现它缺一个关键工具，我就先把 agent 挂到后台，自己把工具装好，再把它切回来继续工作，然后明确告诉它：“我已经替你装好了，现在可以继续。”这样它就能回到更高效的轨道上。

所以这里的重点是：**不要让 agent 在一个“什么都缺、什么都将就”的环境里证明自己。那不是在测试 agent，那是在给 agent 穿小鞋。**

## 二、为什么我认为命令行是目前人与 agent 协作最好的主界面

我接下来讲一个非常明确的观点：**如果你真的想高效地和 agent 协作，你至少要掌握基本的 CLI，也就是命令行。**

为什么？

因为命令行是我目前见过的，**人与 agent 共用成本最低、摩擦最小、组合能力最强**的协作介质。

图形界面当然有它的用途，尤其在看图、看最终结果、做局部交互的时候仍然重要。但是如果你讨论的是“高频、长时、持续协作”，那 GUI 的摩擦通常太大了：

* 需要点来点去；
* 需要窗口切换；
* 需要拖拽上传；
* 需要手动找文件；
* 需要肉眼定位；
* 需要重复做很多低价值操作。

而命令行不是这样。命令行的优点在于，它天然是：

* 文本化的；
* 可组合的；
* 可记录的；
* 可复制的；
* 可版本化的；
* 可自动化的；
* 可被 agent 直接读取和复用的。

所以我现在越来越倾向于把大部分协作都压到命令行和文本系统里去完成。

## 三、文件格式也要为 agent 服务：尽量文本化，尽量低摩擦

和命令行这个观点紧密相关的，是第二个观点：**尽量使用对 agent 友好的文件格式。**

这里我自己的偏好非常明确：

**能用文本，就不用二进制；能用 Markdown/HTML，就尽量少用 Word/PDF。**

为什么这么说？

因为很多二进制格式在今天依然是高摩擦介质。比如 Microsoft Word 文档，或者 PDF。尤其 PDF，我想大家都知道，它虽然是一个很常见的交付格式，但它对自动处理、结构提取、编辑回流、跨工具协作来说，常常都不是最理想的。它经常在“看起来像文档”和“实际上可操作”之间存在巨大的落差。

所以如果目标是让 agent 高效参与，我会建议尽量把中间过程放在：

* Markdown
* HTML
* 纯文本
* 结构化目录

这些格式上。

尤其是 **HTML**，我认为它非常好。HTML 配一个资源文件夹，本质上已经是非常强的交换格式了。它既能承载文本，也能承载图片、样式、音频，展示力强，而且对 agent 来说也比某些封闭格式更容易处理。

如果你进一步考虑即时通讯、跨机器传输、快速分享这类实际需求，那么我很推荐一种做法：**先生成 HTML，再在需要传输的时候把它打包成单文件 HTML。**

也就是把图片、音频等资源尽可能内嵌进去，比如转成 base64 之类，让最终结果成为一个 all-in-one 的 HTML。这样你在 IM 里发、在远端拉、在本地打开，都会轻松很多。你不需要每次传一个文件夹，也不需要担心资源路径丢失。

这类“降低格式摩擦”的小优化，在长时间高频工作里，收益会非常大。

## 四、Git 不是可选项，而是你的长期记忆系统

接下来我想讲一个我认为特别重要、但经常被忽略的东西：**Git。**

Git 不只是一个版本控制工具，它在 agent 时代更像是一个**状态持久化系统**，也是你的**协作记忆系统**。

为什么我会这么看？

因为当 agent 把你的生产效率显著提高之后，新的瓶颈往往不再是“做不做得出来”，而变成了：

* 你记不记得之前做到哪里；
* 你能不能追踪改动；
* 你能不能恢复某个状态；
* 你能不能把进展稳定地保存下来；
* 你能不能在多个尝试之间来回切换；
* 你能不能把自己的上下文负担降下来。

说得直白一点：**以前效率低的时候，你脑子还能勉强记住。现在效率高了，反而更容易跟丢。**

所以 Git 在这里的价值就非常突出。它帮助你：

* 记录过程；
* 固化状态；
* 管理分支；
* 对比改动；
* 回到过去；
* 让 agent 与 agent、人与 agent 之间有共同可追踪的事实基础。

我会把它描述为：**agent 提高了生产速度，Git 负责防止你在高速度里失忆。**

## 五、输入方式也要升级：不要让你的身体成为瓶颈

再讲一个很多人低估的问题：**输入方式。**

如果你开始长时间地和 agent 一起工作，很快就会发现，瓶颈不只是脑力，也包括体力，尤其是手、手腕、手指、肩颈这些非常实际的东西。

我自己有一个很直接的体验：以前我也习惯大量打字，因为环境比较安静，也觉得这样比较可控。但当你真的连续十天、两周，密集地和 agent 长时间协作以后，你会明显感觉到身体在逼近某种物理极限。手指会累，手会发紧，甚至会有过度拉伸的感觉。

所以我现在越来越重视**语音输入**。

在 macOS 上，其实有一些很实用的辅助功能。比如通过辅助功能去开启语音输入，再通过双击 `Fn` 键之类的方式快速唤起。有些模式还允许你先看一下转写结果，修一修，再把它真正送出去。这个流程在某些场景下非常有帮助。

当然，另一种方式就是直接用语音和 GPT 对话。这种高密度语音交互，本身就是一种很大的优势。因为它让你进入一种更连续、更自然、更少身体阻抗的交流模式。

所以我会建议大家把“输入方式”也纳入工作流设计的一部分，不要只想着软件和模型。你需要学习一些**不要求你一直死盯着屏幕**的输入方法。它们的学习成本可能一开始会让你犹豫，但长期看，我认为收益完全值得。

## 六、agent 与 agent 之间怎么共享上下文？我的答案是：Markdown

如果说前面讲的是“人怎么和 agent 协作”，那么接下来我要讲的是“**agent 和 agent 之间怎么协作**”。

我目前最核心的答案很简单：**Markdown。**

为什么还是 Markdown？

因为 Markdown 既足够轻，又足够清晰；既适合人读，也很适合 agent 接续。它在今天非常像一种跨 agent、跨会话、跨机器、跨时间的**共享中介层**。

但我要强调的是：**不是随便写 Markdown 就够了，而是你要有意识地“把 Markdown 写成能被接手的样子”。**

我目前大概会把共享型 Markdown 分成两类。

### 第一类：节省时间的“项目事实记录”

这一类文件记录的是在某次会话、某段 agent 生命周期里，你好不容易摸索出来的关键事实。比如：

* 某个 magic number；
* 某个 URL；
* 某个可用的参数组合；
* 某个环境变量的写法；
* 某个下载镜像；
* 某个坑的绕法；
* 某个实验结论；
* 某个分支或路径的用法。

这类信息非常珍贵，因为它们往往不是“查文档就能立刻知道”的，而是你和 agent 一起踩坑之后换来的。

所以我建议：**不仅要记下来，还要给它加上下文。**

不要只写一个数，不要只贴一个链接。你要说明：

* 它是什么；
* 它为什么重要；
* 它是在什么场景下用的；
* 它有什么限制；
* 以后谁来接手时应该注意什么。

因为如果没有上下文，后来的 agent 看到也未必敢用、会用、能用。

### 第二类：可复用的“内容块”与“技能片段”

还有一种 Markdown，更像是你平时的共享纸条、备忘录、私有便签、常用说明块。它们不一定对应某个项目的唯一事实，但它们是你经常要重复输入、重复说明、重复搬运的内容。

这类东西也非常值得整理。因为它们本质上已经接近“技能”了。你可以把它们做成模板、做成块、做成 skill，在不同 agent 的上下文里调来调去。这样你就不需要反复重打、反复解释、反复从头铺垫。

所以从我的角度看，Markdown 不是“记笔记工具”这么简单，**它是 agent 时代的共享内存。**

## 七、agent 的生命周期管理：后台、前台、恢复、接续

下面讲 agent 的生命周期管理，这也是我觉得特别实用的一块。

### 1. 后台与前台：`Ctrl+Z` 和 `fg`

在实际工作中，我很常用一个操作，就是把正在跑的 agent 先挂到后台。比如在终端里通过 `Ctrl+Z` 之类的方式让它暂时停下，然后你在同一个目录里做一点事，或者在另一个终端处理一点事情，之后再用 `fg` 把它拉回前台继续。

为什么这个方式好？

因为它让你不需要一直通过那种很笨重的 shell escape 去跳来跳去。你可以更自然地和 agent 共用同一个工作现场。

但这里有一个非常重要的注意点：**如果你在 agent 被挂到后台期间，对工作目录做了修改，那么你回来之后一定要明确告诉它。**

这一点非常重要。

因为现在的很多 agent，本质上还是把自己的工作流程理解成一条连续链条，好像默认“它是这个世界里唯一的行动者”。如果它刚刚产出的文件、刚刚改过的分支、刚刚写好的内容，被你悄悄改名了、移动了、重写了，它是会困惑的。

所以最好的做法是：

* 如果改动很大，就直接告诉它你做了什么；
* 如果改动很小，也尽量用简洁但具体的话描述一下；
* 这样能省掉它花很多时间去重新侦查“到底发生了什么”。

### 2. 恢复历史：resume 很重要

另一个我非常常用的功能是 **resume**。

为什么重要？因为它可以让 agent 近乎完整地接回之前的上下文和经验。这对于长任务、隔天继续、暂停后再开工，非常有价值。

但这里也有一个坑：很多系统的会话恢复和目录路径是强相关的。也就是说，如果你把目录挪走了，原来的恢复路径可能就断了，它可能要求你记住一个很长的 hash，恢复起来就很烦。

我自己的一个偏实用的解决方式，是用**软链接**。软链接很轻，不怎么占空间，又能帮助你把原来的路径逻辑保持住。这样旧的 session 编号、恢复入口，也更容易继续使用。

所以我的理解是：**你以为你在“退出”agent，但其实很多时候你并没有真的退出；你只是把它暂时放下，之后仍然可以恢复。**

### 3. 跨机器、跨标签页、跨一周以后：靠 Markdown，不靠侥幸

如果你只是短时间中断，那么 resume 可能就够了。

但如果你讲的是：

* 跨机器；
* 跨环境；
* 跨标签页；
* 甚至隔了一周以后才回来；

那我会说，**这时候真正可靠的仍然是 Markdown。**

因为 session 恢复适合“接着上次那口气继续”，而 Markdown 更适合“把重要内容沉淀成可以重新加载的公共上下文”。

所以简单总结就是：

* **短期接续**：靠 resume；
* **长期共享与跨环境接续**：靠 Markdown。

## 八、怎么跟 agent 说话：不是玄学，而是“像搜索引擎一样说清楚”

接下来我想讲一个经常被神化、但我自己其实越来越去神秘化的问题：**如何向 agent 下指令。**

我的看法是，很多时候，和 agent 说话真的不需要搞得太复杂。某种意义上，**你可以把它当成一种“强化版搜索引擎语言”**。

什么意思？

就是你没必要把每一句都写成非常完整、非常客套、非常铺陈的自然语言长文。很多时候，只要关键词足够准确、方向足够明确，它就能理解你的意图。

比如你要它做同步，你未必要写一大段客套话。你完全可以像扔关键词一样给它：

* 模型名；
* 同步方向；
* 本地目录；
* 远程路径；
* 工具名；
* 限制条件。

它通常就能自己把这些要素拼成可以执行的操作。

但是我要强调一个点：**能省略的是客套和废话，不能省略的是关键关系。**

尤其是：

* 否定词；
* 限制条件；
* 方向信息；
* 前后位置关系；
* from 和 to；
* 输入和输出；
* 源和目标。

这些是不能丢的。

比如做 `rsync` 这种事情，`from A to B` 和 `to A from B` 根本不是一回事。你如果把方向说含糊了，机器不可能凭空替你猜对。它不是你肚子里的蛔虫，它不知道你想的是哪一边是源、哪一边是目标。

所以我的经验是：

**不是要写得很长，而是要写得有结构、有方向、有约束。**

除此之外，不必太迷信什么“高深提示词”。有时候少写一点、准一点，反而更高效。这样不仅省时间，也能降低你的肌肉负担。长期看，这非常重要。

## 九、环境细节真的会决定效率：代理、下载、镜像、Python 环境

下面我讲一组非常实际、非常“脏活累活”、但又特别关键的问题。

### 1. 代理问题：不要让大模型下载把你的代理额度烧光

如果你的 agent 会拉取模型、下载数据、访问 Hugging Face 之类的站点，那你一定要提前想清楚网络路径。

这是我踩过的很现实的坑：很多 agent 默认会直接从 Hugging Face 拉东西，而有些环境下这需要走代理。问题在于，下载大型模型、数据集，动辄就是几个 G、十几个 G，甚至更多。你如果不提前约束它，它很可能会非常老实地把这些大流量统统从代理通道里灌过去。

结果就是：**代理额度被迅速耗尽，速度还慢，钱还多花。**

所以我后来会非常主动地告诉 agent 一些关键信息，比如：

* `no_proxy` 或者等价的无代理环境变量思路；
* 使用镜像站，例如 `hf-mirror.com` 这一类思路；
* 避免把大体积下载走进不必要的代理管道。

而且这里再一次说明，和 agent 交流时，你不需要写成论文。很多时候你只要扔几个关键点：

* no proxy
* no_proxy
* hf mirror
* 某镜像域名

它就能明白你是在提醒它重构下载策略。

这类小设置，往往能同时节省：

* 时间；
* 带宽；
* 代理成本；
* 失败重试成本。

### 2. Python 环境：给它一个干净、现成、可工作的环境

另一个特别值得提前准备的是 Python 环境。

如果你不给 agent 一个合适的环境，它通常会为了“不要污染你的系统”，变得非常保守。它会倾向于自己再套一层 `venv`、或者 `uv`、或者别的隔离方式。这在原则上没有错，但现实问题是：它会增加目录复杂度，增加空间消耗，也增加理解成本。

尤其在磁盘空间本来就紧的时候，这些额外环境很快就会把配额吃掉。最后你会发现，项目还没推进多少，环境倒是套了一层又一层。

所以我自己的建议是：**如果你已经知道某类任务要在一个特定 Python 生态里做，就尽量提前给它一个合适的 conda 环境或者其他你已经整理好的环境。**

这会让它少走很多保守但昂贵的弯路。

---

## 十、agent 特别擅长什么？它尤其适合“数字作品”和“算法复现”

接下来我想讲讲我对 agent 能力边界的一些体会。

我觉得它特别擅长的一类任务，是**创造数字作品**。比如：

* 生成展示文稿；
* 写 Markdown；
* 组织文档；
* 做网页形式的报告；
* 复现实验；
* 重写算法；
* 按要求改造流程；
* 批量处理文本和代码。

这些任务的共同点在于：它们都属于数字世界内部的重组、生成、搬运、整理、复现。agent 在这里非常有优势。

而且命令行环境还给了它一个网页版聊天机器人没有的优势：**你不需要手工“上传一切”。**

在网页里，很多时候你要先想：我要不要传这个文件？那个文件有没有用？这个 PDF 要不要手工挑页？这些准备工作本身就很费精力。

但在命令行和本地目录里不是这样。你可以把相关文件都放在现场，agent 会自己通过：

* grep；
* 文件名；
* 后缀；
* 文件大小；
* 类型过滤；
* 目录结构；

来判断哪些该看、哪些该跳过。它不一定每次都完美，但整体上，这种方式的摩擦小得多。

不过这里也有一个非常关键的提醒：**既然 agent 很依赖搜索和关键词，那么你就要善待它的搜索入口。**

其中最重要的一个入口，就是**文件名**。

不要把文件命名成：

* `1.txt`
* `new.doc`
* `temp-final-final-v2.md`

这种东西对 agent 没有任何帮助。文件名本身就是最廉价、最早暴露给 agent 的提示。

所以请尽量把文件命名成带语义的名字。哪怕你暂时不方便大规模重命名，至少也给目录放一个 README，或者放一个很小的 cheat sheet，告诉它：

* 哪个文件是干什么的；
* 哪个目录是什么内容；
* 哪些是重点；
* 哪些可以忽略。

不要每次都把这些解释塞进 prompt 里。你把它落到文件里，不仅当前 agent 受益，后来的 agent 也会受益。等于是你在帮未来的自己。

## 十一、结果怎么看：本地看、远程看、浏览器看，各有办法

当 agent 开始真正产出结果时，你还需要解决一个非常实际的问题：**怎么高效查看结果。**

比如我个人更喜欢看中文内容，所以我经常会要求它尽量生成中文结果。但问题不只在语言，还在于你怎么打开这些结果。

如果结果在远端服务器上，有几种常见办法。

### 第一种：直接在命令行看

如果只是纯文本、日志、Markdown，命令行直接看就行，这对熟悉 CLI 的人非常自然。

### 第二种：把结果拷回本地再打开

如果是 HTML 报告、包含图片、或者更适合浏览器看的内容，那我常用的方式是先取得绝对路径，再拷回来。

比如你可以先用类似 `readlink -f` 这样的方式拿到绝对路径，再通过 `scp` 之类的方法把它复制回本机，然后直接在本地浏览器里打开。

如果这个 HTML 依赖很多资源文件，那就回到前面说的做法：**尽量把它打包成单文件 HTML。**这样每次只要拉一个文件，不需要同时拉整个资源目录。

### 第三种：在远端起一个 HTTP 服务，再做端口转发

还有一种我也很喜欢的方法，就是在远端目录直接起一个轻量级 HTTP server。比如用 Python 自带的简易方式开一个服务，再把端口通过 SSH 转发到本地。这样你本地浏览器访问本地端口，实际上看到的是远端的 HTML 报告。

这个方式的优点是：

* 不用反复下载；
* 远端一更新，本地刷新就能看；
* 特别适合正在迭代的报告、页面和可视化结果。

为了进一步降低摩擦，我还会建议大家把 SSH 的常用连接写进 `ssh config`，做成短别名。这样你不用每次重打一长串主机信息。再配合 `ssh-copy-id` 这类方式，尽量减少反复输密码。

你会发现，很多真正影响长期效率的，不是那些宏大的技术点，而是这些“小砂子”。真正让你走不动路的，不一定是山，可能只是鞋里的沙子。尤其当你每天和 agent 工作十个小时以上，这些小摩擦会被无限放大。

## 十二、监控不能少：tmux 是朋友，GPU 使用率是指标

接下来我要讲最后一个特别实用的部分：**监控。**

如果你开始同时跑多个任务，或者让 agent 长时间自主运行，那么监控就变得非常重要。

我个人很喜欢用 **tmux**。它真的很适合这种场景。你可以开多个 pane、多个 session，同时看几个地方。

而且你要有意识地监控一些**独立于 agent 自述之外的客观指标**。例如：

* GPU 使用率；
* 显存占用；
* 磁盘空间；
* 下载速度；
* 某个进程是否真的在跑；
* 某个训练是否在 idle；
* 某个循环是否卡住。

为什么说要“独立于 agent 自述之外”？

因为 agent 有时候会很忙，但它“忙”的不一定是你希望它忙的事情。举个最典型的例子：如果你本来希望它调用 GPU，但你通过监控发现 GPU 已经闲了很久，而 agent 还在忙，那这往往意味着它跑偏了，或者退回到了 CPU 路径，或者在做一些低价值的周边事情。

这个时候，有些系统允许你在**不打断主工作流**的情况下，给 agent 加侧边评论。这个能力我特别喜欢。比如你就丢一句非常简短的话：

“GPU idle。”
“GPU 没在用。”
“检查一下是不是没用上 CUDA。”

哪怕是这种很破碎、很简短的语言，它往往也能在下一轮循环里接收到，然后自己去解释、修正、或者调整轨迹。

我觉得这是一个非常好的交互模式。因为它保留了 agent 的自主性，但又没有把你排除在外。你可以在它工作的时候，持续轻量干预。

另外，有些时候你也可以直接问它：

* 当前任务大概还有多久；
* 你现在有没有问题；
* 你卡在哪一步；
* 你下一步准备做什么。

这里我会区分两种交互：

### 一种是与当前工作高度相关的问话

比如 ETA、当前阻塞点、是否需要额外信息。这些直接在原控制台里问就可以，因为它们属于同一条工作线程。

### 另一种是纯检查、纯审阅、纯旁观式的操作

这种我更倾向于开另一个终端，甚至开另一个 agent 去辅助检查。这样不会打断主 agent 的注意力。

某种程度上，这很像你在和同事协作。你不会把一堆无关材料直接摔到正在专心工作的同事面前；你可能会请另一位同事帮你审一眼，或者帮你盯一下状态，再回来给你结论。

当然，多 agent 并行也会带来一个问题：**会话历史管理会变复杂。**尤其是如果多个 agent 混在同一个目录里跑，很容易彼此污染上下文、混淆 session 轨迹。至少以我目前的经验来看，这件事还需要比较谨慎。

所以我现在的态度是：**不要轻易让多个 agent 在同一个工作目录里无序混跑。**如果你要做更严格的隔离，我认为像 `git worktree` 这种思路是很有潜力的，因为它能为不同工作线程提供独立目录，从而隔离 session 历史和文件状态。这个方向我认为值得继续试。

## 十三、agent 最适合什么时候用？尤其适合“你根本不知道从哪开始”的时候

我还想特别讲一点：**agent 最有价值的应用场景，不一定是你已经非常清楚该怎么做的时候，而恰恰是你完全不知道怎么开始的时候。**

这时候，它的价值特别明显。

你可以先和网页聊天机器人聊几轮，拿一个粗略思路；也可以直接把 agent 丢到一台准备好的机器上——有 GPU、有磁盘、有必要工具——然后让它先去探索、先去试、先去开路。

这对“陌生任务”“全新领域”“要先摸路径”的问题，特别有帮助。

但前提仍然是你自己要知道：

* 它擅长什么；
* 它不擅长什么；
* 哪些地方要人盯；
* 哪些地方该预先约束；
* 哪些地方必须给它条件。

换句话说，**你不需要事事亲自做，但你需要做一个合格的环境设计者和过程监督者。**

## 十四、最后一个提醒：不要让 agent 的“初步答案”过早地麻醉你的好奇心

最后我想讲一点更像结尾的话，也是我最近感受特别深的一点。

agent 很强，强在它能迅速给你一个答案、一个方向、一个初步结果。这个能力当然非常有用。但它也有一个潜在风险：**它太擅长给“初步答案”了，以至于你的好奇心有时候会被过早满足。**

你本来可能还应该继续追问、继续深挖、继续做到底，但因为它已经给了一个“看起来差不多”的回答，你的大脑就会产生一种“好像已经知道了”的感觉，于是就停在了预备阶段。

这其实很危险。

所以我现在会提醒自己，也想提醒大家：

**不要让 agent 替你结束探索。**
**不要让初步结果消磨你的兴趣。**
**不要让看起来像答案的东西，提前终结真正的工作。**

在今天，能把事情做出来的人会越来越多。机器人能做，自动化流程能做，agent 也能做。但我越来越觉得，真正稀缺、真正珍贵的，反而是人的那部分东西：

* 你的愿望；
* 你的好奇心；
* 你的不甘心；
* 你的坚持；
* 你想把事情真正做成的那股劲。

因为工具只能放大你已经有的方向，它不会替你生成真正的志向。

所以如果要我用一句话收尾，我会这样说：

**今天，强大的 agent 已经像我们的化身一样，开始替我们在数字世界里行动。越是在这个时候，人最珍贵的不是“会不会按按钮”，而是有没有持续推进的愿望。**
**有愿望的人，会比只有工具的人走得更远。**
**不要停在“差不多”，继续推，继续做，继续把那些过去因为太费劲而被黑箱化的事情，一件一件重新打开。**

谢谢大家。
